We currently have:

  • A Comet server and client (codename “Minerva”) that actually works. Minerva...
    • works over HTTP long-polling, HTTP streaming (optional), and Flash Socket (optional).
    • allows you send a stream of strings in both directions. It works more like TCP than like pub-sub.
    • is completely decoupled from any non-networking code: you're not required to use JSON, XML, RPC, or any client-side GUI stack.
    • works even when the user has many tabs open to your website.
    • ACKs messages in both directions, meaning your messages do not get lost or re-ordered. Most Comet systems get this wrong, usually by lacking ACK logic or having it only in the server->client direction.
    • has robust timeout logic at the transport and Minerva-stream levels, mitigating bad connectivity.
    • is designed to work with the worst HTTP proxies and anti-virus products.
    • loads quickly. Your users don't have to know there's Comet involved.
    • can be completely integrated on the server-side with your own Python code, and on the client-side with your own JavaScript. Even our optional Flash code can be compiled into your own Flash applet. If you don't like Python, you can relay messages to another application server.
    • works over both HTTP and HTTPS (wildcard cert may be required).
    • is tested in old and new versions of IE, Firefox, Chrome, Safari, Opera, and some mobile browsers.
    • doesn't show loading spinners in any browser; you can start a Minerva-stream before the page loads.
    • is written by security-conscious programmers. We review our code when we learn of a new class of attack.
    • Upcoming improvements:
      • Cross-tab connection-sharing, even in browsers without a SharedWorker.
      • Full WebSocket support, after the protocol stabilizes.
  • Extensive server-side programming experience, especially with the Twisted framework.
  • Extensive client-side programming experience, and plenty of custom software to help build large JavaScript applications.
